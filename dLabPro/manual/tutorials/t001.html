Synopsis

Gives a brief introduction to dLabPro and explains some basic concepts like RPN syntax, stack, classes and instances. It also contains information how to run a dLabPro session and how to get help.

Introduction

This tutorial is written to give a short introduction to dLabPro. It shows how dLabPro works and introduces the basic concepts. For detailed information see the reference manual.

dLabPro is an object oriented language which means you will work with instances of classes.

You should be familiar with these two concepts before going on reading this tutorial.

dLabPro classes may have the following types of members:

Methods Procedures belonging to the class, e.g. -status which may print status information on the instance. In dLabPro method identifiers usually start with the character '-'.
Fields Variables of various data types (including class types) belonging the class. The actual values stored in the fields define the internal state of an instance of a class.
Options Options are on/off switches which influence the behaviour of methods. Option identifiers usually start with the character '/'.
Errors Class specific error messages. Error identifiers are four digit numbers starting from 1001.

The first two member types are common to object oriented languages. The latter two are special to dLabPro. You may think of options as command line switches of console programs. Errors just belong to their respective class, meaning for instance there is an error1001 in the class data and an error1001 in the class fst and these two are different errors.

dLabPro is a scripting language which is executed by a command interpreter (so there is no need of compiling and linking code when developing dLabPro programs). The basic interpretation unit is a command which you may simply imagine as one line consisting of interpretable items (called tokens) like identifiers of methods, fields or options, numeric or string constants etc. dLabPro commands end with a semicolon.

Running a dLabPro Session Home Tutorials Top

To start dLabPro open a command shell or terminal and type dlabpro (it is provided that your PATH environment variable includes the location of the dLabPro executable; the command line option --logo makes dLabPro display a version information at start-up)

>dlabpro --logo
      
// dLabPro version 2.5.3 build 1266 win32(ix86,32bit)release PID:3292
dLabPro>_

You are now in the interactive mode and may enter commands. The command prompt consists of the session name (usually "dLabPro") followed by a period and the identifier of the so called active instance. An instance is activated by dLabPro when interpreting its instance name. As nothing has been interpreted in our session so far, the active instance is the dLabPro interpreter itself.

Now you may call a method on the active instance (that is the interpreter). Try -help; and you will get a list of methods of the active instance (that is still the interpreter), classes, options and fields with a short explanation. Try -explain; and you will get brief information on the usage of the active instances class members. After creating an instance of a class (see section working with classes) you can type -help; again and you will get the help table of the current class. If you want to get a description of an identifier (e.g. a method name) use -explain <identifier>:

dLabPro> -explain -status;

----------------------------------------------------------------------
   Explanation of method
   function -status
----------------------------------------------------------------------

   Syntax: <function this> -status

   Prints status information of the function.

----------------------------------------------------------------------

dLabPro>_

So the explanation of -status is: -status is the identifier of a method which belongs to the class function and has the given usage (syntax).

You may end a dLabPro session using the method quit. 

dLabPro> quit;

// Exiting dLabPro...
>_

dLabPro has some command line options. Type

>dlabpro --help

to get a list.

The Syntax Home Tutorials Top

The syntax of dLabPro commands is based on the "reverse polish notation" (RPN), which means that method identifiers/operators have to stand behind the arguments/operands (postfix notation). The following example will clearify what is meant. The expression

2/(3-8)

looks in reverse polish notation like this

2 3 8 - /

This kind of notation is reverse of a prefix notation developed by the Polish mathematician Jan Lukasiewicz in the 1920's. With a few exceptions dLabPro uses this notation.

You can try the simple math example above on the dLabPro command prompt:

dLabPro> 2 3 8 - / -echo;
-0.4dLabPro>_

So the result of the computation is -0.4. Let's now use the method -echo to display the word "hello" in the following way:

dLabPro> "hello\n" -echo; # Push "hello\n" onto the stack and display it
hello
dLabPro>_

As you can see, the argument stands before the method. It is committed to the method through a stack. With "#" you mark the beginning of an inline comment. Block comments are not supported. Whitespaces are not significant. Commands should end with a semicolon and be written in one line (however, line breaks are not significant).

The general syntax of a dLabPro command is:

[<arguments>] <instance> [<options>] <method> [<postfix_arguments>];

<method> stands for the method to be invoked, <instance> stands for the instance which <method> is member of, <arguments> stands for a white space separated list of arguments to <method>, <options> stands for a white space separated list of options to <method> and <postfix_arguments> stands for a white space separated list of postfix arguments to <method>. The command ends with a semicolon. <instance> becomes the active instance when its name is interpreted. It remains active until another instance name is interpreted. So as long as methods have no instance arguments (whose interpretation would let them become the active instance) you may omit <instance>.

While most methods require regular prefix arguments, there are a few methods using postfix arguments (most of them members of the classes function and data). There are also some other exceptions to the RPN syntax in dLabPro. The following table shows an overview:

RPN exceptions
Exception Syntax Example Remark
Creating instances <class> instancename; var i;
Postfix arguments <method> argument; -list scalops;
Formulas :<expression>; :i=i+1; See Tutorial III - Writing Scripts, section "Formulas" for details
The Stack Home Tutorials Top

Arguments are passed to methods or mathematical operations through a stack. There are four types of stack items:

  • boolean values,
  • numbers,
  • strings and
  • instances.

You can display the current stack contents using the method -list: 

dLabPro> 42 "world\n"           # Push 42 and "world\n" onto the stack
dLabPro> -list stack;           # ... and dump stack

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: S "world\n"
                        01: N 42
                 ACTIVE 00: -
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
For more information type -list help;
dLabPro>_

The -list method is one example for non-RPN syntax in dLabPro. It accepts one postfix argument to identify the type of list which shall be displayed. The stack dump shows the two items we pushed on the stack. The most recent item is on top of the stack. The semicolon concluding a command causes dLabPro to clear the stack. Try these two commands: 

dLabPro> 42 "world\n" -list stack; -list stack;

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: S "world\n"
                        01: N 42
                 ACTIVE 00: -
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP --: EMPTY
                 ACTIVE 00: -
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;
dLabPro>_

The first semicolon clears the stack. When -list is called for the second time the stack is empty. Now retype the command omitting the first semicolon. Now the items will remain on the stack:

dLabPro> 42 "world\n" -list stack -list stack;

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: S "world\n"
                        01: N 42
                 ACTIVE 00: -
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: S "world\n"
                        01: N 42
                 ACTIVE 00: -
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;
dLabPro>_

Now let's push all kinds of values:

dLabPro> data x;                   # Create data instance named x
dLabPro> data y;                   # Create another one named y
dLabPro> 1 x 2 3 "x" "y" y TRUE PI # Push some values, do NOT finish the line with a semicolon!
dLabPro.y> -list stack;            # Show stack contents, y is "active" instance

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: N 3.141592653589793
                        01: B TRUE
                        02: I y
                        03: S "y"
                        04: S "x"
                        05: N 3
                        06: N 2
                        07: I x
                        08: N 1
                 ACTIVE 00: y
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;
dLabPro>_        # no "active" instance after semicolon

As we can see, dLabPro tracks the type of values pushed to the stack: "B" stands for boolean, "N" for number, "S" for string and "I" for instance.

After a semicolon the stack is cleared and also the active instance is cleared. So if you want to do something with the active instance after this command, you should omit the semicolon.

Let's now take a closer look on how dLabPro passes arguments to methods through the stacks:

dLabPro> data z        # Create an instance named z of class data
dLabPro.z>"C1" 255 z   # Push some arguments to the stacks and
                       # activate instance z (NO semicolon!)
dLabPro.z> -list stack # Show stack contents (NO semicolon!) NOTE 1)

--------------------------------------------------------------------------------
   Status of instance
   function dLabPro
--------------------------------------------------------------------------------
   (no arguments)
   Program pointer: 0
   Execution mode : 19
--------------------------------------------------------------------------------
             STACK DUMP 00: I z
                        01: N 255
                        02: S "C1"
                 ACTIVE 00: z
                        01: -
                        02: dLabPro
--------------------------------------------------------------------------------
   For more information type -list help;
dLabPro.z> -addcomp   # Call a method (of active instance z)
dLabPro.z> -status    # See what we've got
--------------------------------------------------------------------------------
   Status of instance
   data z
--------------------------------------------------------------------------------
   Components     :      1 (Capacity         : 20)
   Records        :      0 (Capacity         : 0)
   Blocks         :      1 (Records per block: 0)
   Records
   - Length       :    255 bytes
   - Width        :      0 ms
   - Increment    :      0 ms
   - Offset of 1st:      0 ms
   User descriptions
   - 0            :      0
   - 1            :      0
   - 2            :      0
   - 3            :      0
   - 4            :      0
   - File text    : -none-
   - Realization  : -none-
   - Description  : -none-
--------------------------------------------------------------------------------
   Component structure
                        Name  Offset   Bytes   Type
                 0:       C1       0     255    255 char[255]
--------------------------------------------------------------------------------
   To view the data content type z -print;
dLabPro.z>_

Notes:

  • 1) Why can we call the interpreter method -list when instance z is active? Look at the "ACTIVE" list. There are actually three active instances. z is the one with the highest priority. However, if an identifier (as -list) is not member of the first active instance, dLabPro seeks in the second and in the third one. In this case -list is found in the third active instance (the one with the lowest priority). The second active instance position is reserved for variables (see there for details).
  • dLabPro performs reasonable type conversions committing arguments to methods. For instance you could type "C1" "255" z -addcomp; instead of the example above. In this case the second argument will be converted from a string to a number (as required by the method data -addcomp).
  • dLabPro removes the arguments committed through the stack when invoking a method. Type -list stack after the example above and you'll find the stack empty.
  • Some methods or mathematical operations return one or several values. These appear on top of the stack after invoking the method or operation.
  • Postfix arguments do not appear on the stacks.
Numbers and Constants Home Tutorials Top

dLabPro supports complex numbers. The format of complex number literals is <re>+<im>i where the real part <re> and the imaginary part <im> are real number literals including the symbols inf and nan.

Examples:

dLabPro> 1.1 -echo;                # 1.1
dLabPro> -inf -echo;               # -1.7976931348623157081453e+308
dLabPro> i -echo;                  # i
dLabPro> -2.2i -echo;              # -2.2i
dLabPro> infi -echo;               # 1.7976931348623157081453e+308i
dLabPro> 42+0i -echo;              # 42
dLabPro> 1.1+2.2i -echo;           # 1.1+2.2i
dLabPro> -inf+nani -echo;          # -1.7976931348623157081453e+308+nani
dLabPro> -nan -echo;               # nan

Note: On some platforms inf will be represented by the greatest floating point number.

Along with the list of members of the class function (see above) you'll find a list of classes and a list of existing instances. As you can see, dLabPro automatically creates some instances at startup.

Now let's create an instance of the Helloworld-class. The general syntax for instanciating a class is:

<classname> instancename;

So creating a new instance of class Helloworld looks like this:

dLabPro> Helloworld iHello;   # Create an instance named iHello of class Helloworld

By convention instance names should begin with 'i' followed by a capital letter. The usual exceptions apply: simple names like i, j, k for variables and x, y, z for data tables are acceptable.

This help table shows the members of the Helloworld-class. Remember that method identifiers usually start with '-' and option identifiers with '/'. As you can see Helloworld has got one method (-sayhello) which we'll use now:

dLabPro> iHello -sayhello;  # Invoke method -sayhello


Hello World!

dLabPro> iHello /german -sayhello # Invoke the same method nut using the option /german

Switch to German...        # Printed by /german

To manipulate the displayed text you can change the content of the field greeting_english with the method -set (of the interpreter) like this:

dLabPro> "hi!" iHello -set greeting_english;
dLabPro> iHello -sayhello


hi!

Remember the basic structure of dLabPro commands (see section "The syntax" above):

[<arguments>] <instance> [<options>] <method> [<postfix_arguments>];

The first token ("hi!") is an argument to the -set-method. It will replace the content of the field greeting_english. The second token specifies the instance for which -set shall be invoked. Behind the arguments follows the method identifier (-set) followed by the identifier of the field which is a postfix argument to -set. Remeber that even though the instance iHello is active, -set (which is a method of the function-class) may be called (for details see above).

Let's, finally, have a look at the end of an instances life cycle: its destruction. To destroy an instance, use the -destroy method of the function-class:

dLabPro.iHello> iHello -destroy;        # Delete instance iHello
dLabPro> iHello                         # What happened to iHello?
